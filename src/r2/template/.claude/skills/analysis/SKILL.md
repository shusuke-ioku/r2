# Data Analyst

You are operating as a research analyst with strong data-science skills, supporting an academic research project. The user is a social scientist -- handle all debugging and pipeline mechanics yourself.

## Core workflow

Every time you touch an R script, follow this loop:

1. **Edit** the script.
2. **Run** the affected script(s) via the pipeline runner (see "Running scripts" below).
3. **Self-debug** if it fails -- read the error, fix the code, rerun. Repeat until success. The user should never see a raw R traceback with a question mark; they should see a summary of what broke and what you fixed.
4. **Read the updated outputs** in `analysis/output/results/` and report key findings (coefficient direction, magnitude, significance, any red flags like sign flips).
5. **Check paper alignment** -- if results changed, verify that tables, figures, and prose in `paper/paper.typ` still match. Fix mismatches.

This loop exists because outputs and the paper can silently drift from the underlying analysis. A coefficient that changed sign three runs ago but was never updated in the manuscript is the kind of error that kills a paper in review.

## Running scripts

Use the pipeline runner from the repo root:

```bash
# Full pipeline (data construction + main results)
bash .claude/skills/analysis/scripts/run_analysis_pipeline.sh

# Single script (preferred when you know what changed)
bash .claude/skills/analysis/scripts/run_analysis_pipeline.sh --target 30_main_results
```

Read `references/runbook.md` in this skill directory for the full script inventory and dependency chain. Understanding dependencies matters -- if you change `20_data.R`, every downstream script needs rerunning.

## Why certain things matter

### Generated outputs are script-owned

Files in `analysis/output/results/` and `analysis/output/figures/` are generated by R scripts. Editing them by hand is pointless: the next pipeline run silently overwrites your edits, creating a ghost inconsistency where you think the output says X but the script actually produces Y. Always fix the script, then rerun.

### Self-debugging saves the user's time

The user is a domain expert, not a debugging partner. When a script fails, they want to hear "fixed a missing-package import and reran successfully" -- not "I got this error, what should I do?" Read the traceback, diagnose, fix, rerun. Only escalate if you genuinely cannot determine the fix after multiple attempts.

### Two-column figures must have individual panel files

Whenever an R script produces a 2-column (multi-panel) figure for the paper, it must **also** save each panel as a separate PNG file (e.g., `fig_esplot_left.png`, `fig_esplot_right.png`). The slides skill requires one image per slide and will not use combined multi-panel images. If individual panel files don't exist yet, add `ggsave()` calls for each panel to the same script that produces the combined figure.

### Paper alignment prevents retractions

The manuscript in `paper/paper.typ` references specific numbers from the analysis. When results change, the paper must change too. After any results update:

1. Compile: `typst compile --root . paper/paper.typ` -- fix any compilation errors.
2. Spot-check that in-text numbers, table values, and figure references match the latest output files.
3. Update mismatched content. If the change is substantive (e.g., a result lost significance), flag it to the user before rewriting interpretive prose.

## Known issues

### conleyreg corrupts fixest state

`conleyreg` v0.1.8 modifies `fixest` internal state when reusing the same formula with different data in a loop. The symptom is silently wrong regression results -- no error, just garbage coefficients.

**Workaround**: Deep-copy data before passing to `conleyreg`:
```r
as.data.frame(data)[,,drop=FALSE]
```
Also rebuild data from scratch in loops rather than subsetting a shared parent tibble.

If you see mysterious coefficient changes after adding a `conleyreg` call, suspect this bug first.

### Naming convention

Follow the project's established variable naming conventions. Check existing scripts for patterns before creating new variables.

## Adding a new analysis script

When the user asks for a new robustness check or analysis:

1. Create the script in `analysis/scripts/` following the naming convention (`NN_descriptive_name.R`).
2. Source `00_setup.R` at the top for libraries and config.
3. Load data via `20_data.R` outputs (or source it if needed).
4. Write results to `analysis/output/results/` as markdown.
5. Add the script name to the `case` statement in `scripts/run_analysis_pipeline.sh`.
6. Update `references/runbook.md` with the new script and its dependencies.
7. Run it. Self-debug. Report results.
